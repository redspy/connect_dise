# 그림 릴레이 (Relay Drawing) — 게임 기획

## 컨셉

스마트폰을 스케치북 삼아 제시어를 그리고 맞히는 왁자지껄 파티 게임.
1명의 그림을 다음 사람이 보고 단어로 설명하고, 그 단어를 또 다음 사람이 그리는 방식으로 릴레이가 진행됩니다. 마지막에 PC 화면을 통해 처음 제시어가 어떻게 변질되었는지 확인하는 것이 핵심 재미 요소입니다. (Gartic Phone 스타일)

- **참가 인원**: 2~8명 (권장 4명 이상)
- **플레이 타임**: 라운드당 약 5~10분

---

## 게임 흐름

```
로비 → QR 스캔 → 닉네임 입력 및 캐릭터 설정 → 전원 준비
→ 게임 시작 (라운드 수 및 제한 시간 설정 확인)
→ [턴 1] PC: "각자의 폰을 확인하세요" / 모바일: 각자에게 제시어 할당 → 그림 그리기 (제한시간)
→ [턴 2] 모바일: 이전 사람이 그린 그림 확인 → 단어로 설명하기 (제한시간)
→ [턴 3] 모바일: 이전 사람이 적은 단어 확인 → 다시 그림 그리기 (제한시간)
→ ... (설정된 턴 수만큼 반복)
→ [결과 발표] PC 화면에서 각 플레이어의 릴레이 결과물을 순차적으로 공개하며 다같이 감상
→ 다시하기 / 로비
```

---

## 핵심 규칙

1. **홀수 인원 vs 짝수 인원**:
   - 마지막 턴이 항상 '그림'으로 끝나거나 '단어'로 끝나도록 턴 수를 조절할 수 있습니다 (기본적으로 인원수와 동일한 턴 진행).
2. **턴 교차**:
   - `단어 제시` -> `그림 그리기` -> `단어 맞히기` -> `그림 그리기` 순서로 무조건 번갈아 진행됩니다.
3. **랜덤 셔플**:
   - 각 턴이 끝날 때마다 결과물이 무작위 다른 사람에게 전달됩니다 (단, 자신의 체인으로 돌아오지 않도록 라우팅 처리 필요).
4. **제한 시간**:
   - 턴마다 제한 시간이 있으며 (예: 그림 60초, 단어 30초), 시간이 끝나면 강제로 현재 상태가 제출됩니다.

## 화면 구성

### 1. 호스트(PC) 화면

- **로비 대기실 (Lobby)**
  - 참가자 목록 및 프로필 (그림을 그릴 수 있는 캔버스 아바타 지원 시 재미 UP)
  - 게임 설정: 라운드 수 (인원수 맞춤/고정), 제한 시간(짧게/보통/길게)
  - 시작 버튼
- **게임 진행 중 (In-game)**
  - 신나는 배경음악 + 남은 제한 시간(타이머) 크게 표시
  - 누가 제출을 완료했는지 상태 표시 (체크마크)
  - "폰 화면에 집중해주세요!" 등 안내문
- **결과 발표 (Result / Replay)**
  - **가장 중요한 화면**: 한 플레이어의 주제가 어떻게 변해왔는지 슬라이드쇼 형태로 1-Step씩 애니메이션과 함께 극적으로 공개
  - 공개 중 효과음(박수, 웃음) 및 이모티콘 리액션 (모바일에서 버튼 누르면 PC 화면에 말풍선/이모티콘 팝업)

### 2. 모바일(컨트롤러) 화면

- **대기실 (Lobby)**
  - 닉네임 입력란 (+ 터치로 직접 그리는 프로필 아바타 기능 있으면 좋음)
  - 준비 완료 버튼
- **그림 그리기 턴 (Draw)**
  - 상단: "제시어: [사과를 먹는 기린]" 또는 "[이전 사람의 단어]"
  - 중앙: 터치 캔버스 영역 (Fabric.js 또는 순수 Canvas API 사용)
  - 하단: 간단한 도구 모음 (펜 색상 4~5개, 지우개, 전체 지우기 기능, 굵기 조절)
  - 상단 코너: 제한 시간 진행바
  - 완료 버튼 (시간 종료 시 자동 제출)
- **단어 맞히기 턴 (Guess)**
  - 상단: 이전 사람이 그린 그림 이미지 (모바일에 꽉 차게)
  - 하단: 정답/설명 텍스트 입력 폼 (키보드 팝업 영역 고려해서 UI 상단 배치)
  - 제출 버튼
- **결과 관전 (Spectate)**
  - "PC 화면을 보세요!" 안내 메시지
  - (선택) 리액션 버튼 3~4개 (👍, 😂, 😲, ❓) - 누르면 PC 화면에 날아감

## 기술적 고려사항 및 구조

### 1. 데이터 흐름 및 저장 (SessionManager / GameLogic)
- 이 게임은 실시간 액션보다는 **상태(State)와 턴(Turn) 동기화**가 핵심입니다.
- **스토리 체인(Story Chain)** 데이터 구조:
  ```json
  [
    {
      "originalAuthor": "PlayerA",
      "steps": [
        { "type": "word", "content": "로봇 청소기를 타는 고양이", "author": "PlayerA" },
        { "type": "draw", "content": "data:image/png;base64,...", "author": "PlayerB" },
        { "type": "word", "content": "루돌프를 타는 산타", "author": "PlayerC" },
        { "type": "draw", "content": "data:image/png...", "author": "PlayerD" }
      ]
    },
    // ... 다른 플레이어들의 체인
  ]
  ```
- 이미지 데이터는 Base64 문자열로 소켓을 통해 전송합니다. 캔버스 크기를 고정(예: 800x600 혹은 모바일 해상도 비율)하고 압축률을 조정하여 패킷 크기를 최적화해야 합니다.

### 2. 캔버스 드로잉 (Frontend)
- **모바일 캔버스**: Touch 이벤트(`touchstart`, `touchmove`, `touchend`)에 대응하는 부드러운 드로잉 로직 필요.
- 베지어 곡선 보간 등을 적용하면 손가락으로 대충 그려도 선이 예쁘게 나옵니다.
- 지우개 기능은 `globalCompositeOperation = 'destination-out'` 활용.

### 3. 메시지 프로토콜

| 방향 | type | payload | 설명 |
|------|------|---------|------|
| H→All | `roundStart` | `{ turnType: 'draw'|'word', timeLimit, content: '...', author: '...' }` | 각자에게 다른 content 분기 전송(유니캐스트 권장) |
| M→H | `submitTurn` | `{ type: 'draw'|'word', content: '데이터' }` | 모바일에서 결과물 제출 |
| H→M | `turnStatus` | `{ completedPlayers: [...] }` | 누가 완료했는지 진행 상황 업데이트 |
| H→All | `showResults` | `{ storyChains: [...] }` | 모든 턴 종료 후 결과 발표 시작 |
| M→H | `sendReaction` | `{ emoji: '😂' }` | 결과 발표 중 모바일에서 리액션 발송 |

## 개발 마일스톤 (구현 순서)

1. **캔버스 프로토타입**: 모바일 화면에서 부드럽게 그림이 그려지고, 선 색상/굵기 변경 및 지우개가 작동하는지 테스트 (Base64 변환 후 이미지태그로 잘 렌더링 되는지 확인).
2. **턴 관리 시스템**: 호스트가 N명의 플레이어에게 각각 다른 주제를 할당하고, 타이머를 돌리며, 모두가 제출했을 때 다음 사람에게 올바르게 데이터를 섞어 넘겨주는(Routing) 로직 구현.
3. **UI 연동**: 상태에 따라 모바일 화면이 `단어 입력 뷰` <-> `그림 그리기 뷰`로 올바르게 전환되는지 연동.
4. **슬라이드쇼 연출**: 호스트 화면의 결과 발표 연출. 스토리가 공개될 때 타이밍과 애니메이션(CSS Transition) 작업.
5. **폴리싱**: 배경음악, BGM, 리액션 이모티콘, 예쁜 폰트 적용.

이 기획안을 바탕으로 바로 호스트/모바일 기본 뼈대 코드 작성 및 캔버스 구현을 시작할 수 있습니다.
