# 🤖 connect_dise 멀티 에이전트 게임 디자인 & 선정 프로세스 가이드

본 가이드는 `connect_dise` 플랫폼의 콘텐츠를 확장할 때, 각 분야의 전문적인 페르소나를 주입받은 AI 에이전트들이 유기적으로 협업하여 게임을 선정하고 기획안을 도출하는 **7인 확장 에이전트 협업 모델**의 운영 표준을 정의합니다.

---

## 👥 1. 7인 에이전트 역할 및 페르소나 정의

새로운 게임 아이템을 균형 있고 다각적으로 검증하기 위해 각기 다른 핵심 임무와 페르소나를 가진 에이전트들을 구성합니다.

| 에이전트명 | 담당 분야 / 페르소나 | 주요 평가 기준 |
| :--- | :--- | :--- |

| **`OnlineGameProducer`** | **온라인 게임 기획자**<br>대규모 실시간 멀티플레이어 기획 및 비대칭 소셜 게임 전문가 | 소셜 경쟁 요소, 매끄러운 턴/매치메이킹 흐름, 플레이어 몰입 유도 및 다운타임 제거 |

| **`SeniorGameDev`** | **시니어 게임 개발 엔지니어**<br>다수 상용 게임 빌드 및 코드 재사용성 극대화 전문가 | 개발 공수(속도), 플랫폼 SDK 공유 모듈 재활용성, 기존 게임 템플릿 코드 재사용 비중, 코드 모듈화 |
| **`PMOrchestrator`** | **프로젝트 매니저 (PM)**<br>프로덕트 오너 및 회의 주재자 | 각 분과 검토 내용 조율, 최종 의사결정(게임 선정), 상세 기획/구현 명세서 도출 |

## 🔄 2. 단계별 프로세스 워크플로우

```
[1단계: 아이디어 탐색] ➔ GameScout & OnlineGameProducer가 협업하여 3선 후보 제안
       ↓
[2단계: 다면 교차 검증] ➔ 4대 분과(아동/UIUX/기술/개발) 에이전트가 각 후보군 심층 평가
       ↓
[3단계: 가상 의사결정 회의] ➔ PMOrchestrator가 7자 토론을 중재하여 최적의 1개 아이템 최종 낙점
       ↓
[4단계: 연동 명세 문서화] ➔ PMOrchestrator가 폴더 구조, 소켓 API 프로토콜, 스켈레톤 명세서 도출
```

### 1단계: 아이디어 탐색 및 리서치
- **`GameScout`**과 **`OnlineGameProducer`**는 웹 검색을 활용하여 대립 개념, 동조, 실시간 협력 등 멀티 디바이스 환경에 특화된 3가지 후보 게임 모델을 설계합니다.

### 2단계: 4대 분과 검토 보고서 작성
- 후보 게임들에 대해 4개 분과 에이전트가 동시에 개별 검토를 구동합니다.
  - **아동 검증**: 규칙 이해(60초 내), 텍스트 최소화, 아동 눈높이의 카툰 테마 제안.
  - **사용성 검증**: 모바일 소프트 키보드 회피 방안, 스크롤바가 필요 없는 fixed viewport, 대각 드래그 햅틱 휠 설계.
  - **기술 아키텍처**: 실시간 WebRTC P2P 스트리밍 비용 산정, 소켓 통신 병목 및 AFK(Timer Expiry) 방지책 수립.
  - **개발 효율성**: 기존 플랫폼 코드(`hidden-agent`, `relay-drawing` 등) 중 재사용할 부분과 모듈화 범위 특정.

### 3단계: 가상 7자 보드 미팅 (Debate)
- **`PMOrchestrator`**는 각 파트의 분석 리포트를 기반으로 토론을 주재합니다. 기술 구현 난이도, 모바일 사용성, 아동 접근성, 기획의 재미 요소를 고루 비교하여 합리적인 절충안을 만듭니다. (예: 마피아 전리품 분배를 해적 보물 테마로 피벗).

### 4단계: 명세서 도출
- 선정한 게임에 대해 신규 파일 경로, 레지스트리/빌드 번들링 구성, 상세 소켓/P2P API 프로토콜, 딜레마 정산 공식, 에러 복구 전략을 명시한 기획 보고서를 아티팩트로 출력합니다.

## 🏆 3. 성공적인 프로세스 운영을 위한 가이드라인

1. **상태 관리 철저**: 게임 상태(State)의 소유권은 반드시 Host에 둡니다. 모바일 클라이언트는 전달받은 상태를 단순히 렌더링하고, 입력 행위만 송신하도록 설계해야 재연결 관리가 쉽습니다.
2. **모바일 키보드 간섭 최소화**: 아동 플레이 및 화면 깨짐을 원천적으로 막기 위해 텍스트 입력 대신 숫자패드 팝업이나 드래그 칩, 선택 단추 중심의 UI 기획을 지향합니다.
3. **P2P와 Socket.IO의 역할 분배**: 실시간 자이로나 드로잉 좌표 스트림은 P2P 채널로 분산하고, 턴 최종 제출 및 정산 정보는 유실 방지를 위해 신뢰성 높은 Socket.IO로 라우팅합니다.
4. **테마의 캐주얼화**: 폭력적이거나 복잡한 성인 위주 테마보다는 직관적이고 경쾌한 애니메이션 테마(예: 해적, 동물, 우주선 등)를 접목하여 플레이 허들을 낮춥니다.
5. **로비 재연결 화면 프리징 방지**: 새로고침이나 연결 복원 시 모바일에서 로딩화면을 띄우는 구조라면, 호스트는 `lobby`나 `loading` 페이즈 상태일 때도 재접속(`onPlayerRejoin`) 신호에 반응하여 반드시 로비 상태 패킷(예: `lobbyState`)을 송신(Unicast)함으로써 모바일의 로딩 상태를 해제하고 로비 대기 화면이나 프로필 화면으로 무사히 복구되게 해야 합니다.
6. **공통 `.hidden` 스타일 명시**: 모든 호스트/모바일 화면에서 공통적으로 사용할 `.hidden { display: none !important; }`를 각각의 CSS 파일에 누락 없이 지정하여, dynamic classList toggle 동작 시 엘리먼트 렌더링 충돌을 방지합니다.
7. **호스트 컨테이너 레이아웃 겹침 방지**: 공용 화면의 가상 felt 백그라운드 요소들이 `z-index: 0~1`로 그려지기 때문에, 호스트 게임판의 메인 껍데기 컨테이너에는 반드시 `position: relative; z-index: 10;` 스타일링을 명시하여 콘텐츠가 배경 이미지 뒤로 가려지는 오류를 사전에 방지합니다.
8. **로컬 데모 모드 탑재**: 신규 게임 개발 시 타 기기 연동 없이 호스트 단독 화면에서 게임을 관찰/시연할 수 있는 데모 모드(`DemoSimulator.js`)를 반드시 구현하여, 로비 화면 내 버튼 클릭을 통해 가상 플레이어들이 퀴즈 제출, 투표, 라운드 클리어 등을 자동으로 플레이하는 시뮬레이션을 내장합니다.

## 💾 4. 산출물 아카이빙 및 저장 표준 (Deliverable Archiving Standard)

AI 세션 내부에서 수립된 임시 기획 아티팩트가 유실되지 않고 리포지토리에 영구적으로 버전 제어되도록, 개발 완료 시점 또는 주요 체크포인트마다 다음 규격에 맞추어 산출물을 아카이빙합니다.

### 📁 아카이브 디렉토리 구조
각 게임마다 `docs/games/<game-id>/` 경로 하단에 아래의 고정된 파일명으로 이관 저장해야 합니다.

```
docs/games/<game-id>/
├── spec.md                   # 게임 설계 명세 (임시 파일: implementation_plan.md)
├── test-report.md            # E2E 테스트 및 검증 결과서 (임시 파일: walkthrough.md)
├── task-checklist.md         # 개발 마일스톤 및 체크리스트 (임시 파일: task.md)
└── retrospective-meeting.md  # 7인 에이전트 미팅 및 리서치 회고록 (임시 파일: <game_id>_selection_report.md)
```

### ⚡ 자동 아카이빙 도구 사용법
리포지토리 루트에서 다음 Node.js 스크립트를 실행하여 세션 임시 산출물들을 원클릭으로 가공 복사할 수 있습니다:

```bash
# 사용법:
# node scripts/archive-artifacts.js <game-id> <conversation-id> [app-data-dir]

# 예시:
node scripts/archive-artifacts.js pirate-plunder 0ef4c056-cfcf-43e9-9550-f3e89a6917ec
```

이 단계는 **배포 및 코드 커밋 전 필수 수행 항목**으로 정의하며, 프로세스 진행 중 생성된 모든 산출물이 유실 없이 저장소에 축적되도록 보장합니다.
